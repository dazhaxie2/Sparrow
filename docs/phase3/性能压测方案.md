# Phase 3 数据库性能与压测方案

> 目标:10M 用户 · 峰值 1 万 QPS(ROADMAP Phase 3 验收线 `ROADMAP.md:138`)
>
> 本方案在 Phase 2(1000 QPS / 10M 用户)基础上,把负载发生器从单进程 Node 换成 **k6**,并把基础设施天花板抬到 10k QPS 包络。

## 0. 与 Phase 2 的差异

| 维度 | Phase 2 | Phase 3(本方案) |
|---|---|---|
| 负载引擎 | `phase2-load.mjs`(单进程 Node,setInterval tick) | `k6-phase3.js`(k6,`ramping-arrival-rate` 开放模型) |
| 引擎上限 | ~2–3k 出站 RPS 即触顶 | 单机轻松 10k+ RPS |
| 目标 QPS | 1000 | 10000(阶梯 2k→5k→10k) |
| MySQL | `max_connections=500` / `buffer_pool=1G` | `max_connections=1500` / `buffer_pool=4G` |
| Redis | `maxmemory=256mb` | `maxmemory=1gb` + io-threads |
| HikariCP | pool 30 / min-idle 10 | pool 50 / min-idle 20 |
| Lettuce | pool 50 | pool 100 |
| JVM `-Xmx` | 128–384m | 768m(ai 1g) |
| 容器资源 | 无限制 | deploy.resources.limits |
| token 池 | 10000 | 50000 |

**关键认知**:万级 QPS 在 ROADMAP 里是 **Phase 3**(`ROADMAP.md:26,138`),而 Phase 3 的真正解法 —— ShardingSphere 分库分表、读写分离、Kafka 集群、多实例 —— **均未实现**。本方案只做「单实例 + 抬天花板」能撑到的极限,定位当前架构的真实拐点,不实现分片。

## 1. 容量分析

### 1.1 数据规模

| 表 | 行数 | 存储估算 | 说明 |
|---|---:|---:|---|
| `sparrow_user.t_user` | 1000 万 | ~2 GB | 10M 行 × ~200B/行(含索引) |
| `sparrow_trade.t_order` | < 100 万 | ~200 MB | 订单量远小于用户量 |
| `sparrow_graph.tech_node` | ~1000 | < 1 MB | 维基级扩容后千级节点 |
| `sparrow_graph.tech_edge` | ~5000 | < 1 MB | 依赖边 |
| Redis(token + cache) | ~50 万 key | ~500 MB | 5万活跃 token + 图谱缓存,1GB 上限留余量 |

### 1.2 流量分解(10k QPS,端点权重同 Phase 2)

| 端点 | 占比 | QPS | 数据源 | 预期延迟 |
|------|---:|---:|--------|---------|
| `GET /api/graph/tree` | 40% | 4000 | Redis(95%+ 命中) | < 10ms |
| `GET /api/graph/overview` | 15% | 1500 | Redis | < 10ms |
| `GET /api/graph/search` | 10% | 1000 | MySQL | < 50ms |
| `GET /api/graph/node/{id}` | 10% | 1000 | Neo4j | < 30ms |
| `GET /api/graph/nodes` | 8% | 800 | MySQL | < 50ms |
| `POST /api/user/login` | 7% | 700 | MySQL + Redis | < 50ms |
| `GET /api/trade/products` | 5% | 500 | 静态 | < 5ms |
| `GET /api/trade/orders` | 3% | 300 | MySQL(鉴权) | < 50ms |
| `POST /api/ai/ask` | 2% | 200 | LLM(Sentinel 限流) | 429 拒绝大部分 |

**实际 DB 负载** ≈ 1000 + 800 + 700 + 300 ≈ **2800 QPS**(Phase 2 是 280 QPS,放大 10 倍)。

⚠️ Redis 命中率是命脉:tree/overview 共 5500 QPS 全压 Redis,若命中率从 95% 跌到 80%,回源到 Neo4j 的全树扫描会瞬间打爆。

## 2. 基础设施调优(本次改动)

### 2.1 MySQL(`backend/docker/mysql/my.cnf`)

| 参数 | Phase 2 | Phase 3 | 理由 |
|------|---|---|------|
| `max_connections` | 500 | 1500 | 多服务高并发连接余量 |
| `innodb_buffer_pool_size` | 1G | 4G | 10M 用户索引热数据 |
| `innodb_buffer_pool_instances` | 4 | 8 | 减少缓冲池锁竞争 |
| `innodb_log_file_size` | 512M | 1G | 高写入下减少 checkpoint |
| `innodb_io_capacity` | — | 2000 | SSD IOPS 余量 |
| `innodb_io_capacity_max` | — | 4000 | 突发 IOPS |
| `thread_cache_size` | — | 100 | 避免频繁建/销线程 |
| `table_open_cache` | — | 4000 | 热表并发打开 |
| `max_prepared_stmt_count` | — | 65535 | 配合 JDBC cachePrepStmts |

### 2.2 Redis(`backend/docker/redis/redis.conf`)

| 参数 | Phase 2 | Phase 3 | 理由 |
|------|---|---|------|
| `maxmemory` | 256mb | 1gb | 5万 token 热集 + 图谱缓存 |
| `tcp-backlog` | 511(默认) | 2048 | 10k QPS SYN 积压 |
| `io-threads` | — | 2 | 多线程 I/O 降单线程瓶颈 |
| `io-threads-do-reads` | — | yes | 读侧也走多线程 |

### 2.3 连接池(`deploy/nacos/sparrow-common.yaml`)

| 参数 | Phase 2 | Phase 3 |
|------|---|---|
| HikariCP `maximum-pool-size` | 30 | 50 |
| HikariCP `minimum-idle` | 10 | 20 |
| Lettuce `max-active` | 50 | 100 |
| Lettuce `max-idle` | 20 | 40 |
| Lettuce `min-idle` | 10 | 20 |

> 5 服务 ×50 = 250 连接 < MySQL `max_connections=1500`,余量充足。

### 2.4 JVM(`backend/Dockerfile`)

| 服务 | Phase 2 `-Xmx` | Phase 3 `-Xmx` |
|---|---|---|
| gateway / user / graph / trade | 256m | 768m(`-Xms384m`) |
| ai | 384m | 1g(`-Xms512m`) |

### 2.5 容器资源(`docker-compose.yml`)

| 服务 | cpus | memory |
|---|---|---|
| mysql | 4.0 | 6G |
| redis | 2.0 | 1500M |
| neo4j | 2.0 | 2G |

## 3. 负载引擎:k6

### 3.1 为什么换 k6

- Node 单进程 `setInterval` tick 模型在 ~2–3k 出站 RPS 触顶(单线程 JSON 解析 + OS socket)。
- k6 是 Go 实现,`ramping-arrival-rate` 开放模型按目标 RPS 生成虚拟用户,单机 10k+ RPS 轻松达成,自带 p50/p95/p99 与 RPS 指标。

### 3.2 安装

```powershell
# 方式一:winget(推荐)
winget install grafana.k6

# 方式二:choco
choco install k6

# 方式三:不装,k6-load.ps1 自动回退到 docker run grafana/k6
```

### 3.3 脚本

| 文件 | 用途 |
|------|------|
| `backend/scripts/k6-phase3.js` | k6 主脚本,9 端点加权随机,环境变量驱动 |
| `backend/scripts/k6-load.ps1` | PS 包装器,探测 k6 / 回退 docker,校验 token,自动改写 BASE_URL |

环境变量:`BASE_URL` / `TARGET_QPS` / `DURATION` / `RAMP_UP` / `P99_LIMIT` / `TOKENS_FILE`。

阈值:`http_req_failed < 1%`(429 单列不计入失败)、`p(99) < P99_LIMIT`。

## 4. 执行手册

### 4.0 前置:全栈拉起 + 推 Nacos

```powershell
docker compose up -d --build
# 等 30s 让健康检查通过,然后推送抬升后的 Nacos 配置
powershell -File deploy/nacos/load.ps1
```

### 4.1 Stage A — 小样本验证(打通链路)

目的:用最小成本验证 k6 引擎、token 池、调优配置无误,行为对齐 Phase 2。

```powershell
# 1. 灌 10 万用户(约 30s)
docker exec -i sparrow-mysql mysql -uroot -proot123 sparrow_user -e "SET @SEED_TOTAL=100000; SOURCE /dev/stdin;" < backend/scripts/seed-users.sql
# 或直接进容器:SET @SEED_TOTAL=100000; SOURCE 路径;

# 2. 收 1000 token(约 1 分钟)
powershell -File backend/scripts/seed-tokens.ps1 -Count 1000

# 3. k6 跑 1000 QPS / 60s,P99 300ms(对齐 Phase 2 验收线)
powershell -File backend/scripts/k6-load.ps1 -Qps 1000 -DurationSeconds 60 -P99Ms 300
```

**期望**:PASS。与 Phase 2 输出对齐,确认 k6 引擎与 token 池工作正常。

### 4.2 Stage B — 全量(10M 用户 / 10k QPS)

```powershell
# 1. 灌满 10M 用户(5-15 分钟,默认 @SEED_TOTAL=10000000)
docker exec -i sparrow-mysql mysql -uroot -proot123 sparrow_user < backend/scripts/seed-users.sql

# 2. 收 5 万 token(约 15-20 分钟)
powershell -File backend/scripts/seed-tokens.ps1 -Count 50000 -Concurrency 50

# 3. 阶梯加压,各 60s,记录每档 p50/p95/p99、错误率、429 数
powershell -File backend/scripts/k6-load.ps1 -Qps 2000  -DurationSeconds 60 -P99Ms 500 -RampUp 10
powershell -File backend/scripts/k6-load.ps1 -Qps 5000  -DurationSeconds 60 -P99Ms 500 -RampUp 10
powershell -File backend/scripts/k6-load.ps1 -Qps 10000 -DurationSeconds 60 -P99Ms 500 -RampUp 15
```

## 5. 预期瓶颈(本次不会修复,如实记录)

| 瓶颈 | 触发点 | Phase 3 真正解法(超出本次范围) |
|------|--------|------|
| **AI 端点 Sentinel 限流** | Nacos `sparrow-ai-flow-rules` `count:5`,2% × 10k = 200 QPS-in → 绝大多数 429 | 提升限流阈值 / 队列化 / 语义缓存(Phase 3 规划) |
| **单 MySQL 容器** | 2800 QPS 实际 DB 负载逼近单机上限 | ShardingSphere 16库×64表 + 读写分离(`ROADMAP.md:115`) |
| **单 Kafka broker(RF=1)** | 事件链路无冗余 | Kafka 集群 RF=3(`ROADMAP.md:116`) |
| **单服务实例** | gateway/user/graph/trade/ai 各 1 个 | 多实例 + 负载均衡(K8s HPA,Phase 4) |
| **Redis 命中率** | tree/overview 5500 QPS 全压 Redis,命中率掉则 Neo4j 全树扫描 | Singleflight 防击穿 + 多级缓存 |
| **会员状态 Feign 同步调用** | `nodeDetail()` premium 节点每次同步查 user 服务 | Redis 缓存会员状态(30s TTL) |
