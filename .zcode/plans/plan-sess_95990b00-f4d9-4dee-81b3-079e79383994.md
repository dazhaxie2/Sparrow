# 单机资源优化 + 虚拟线程升级计划

## 现状摘要（摸底结论）

- **服务器**：16G 内存 / 8 核+ / Windows self-hosted runner，全栈 16 容器实测 ~9G RSS
- **JDK**：17（硬编码在 7 处：`pom.xml` + `mvn17.ps1` + Dockerfile 6 个 runtime + 1 个 build stage）
- **Spring Boot**：3.3.5（**已原生支持** `spring.threads.virtual.enabled=true`，无需升级）
- **依赖兼容性**：MyBatis-Plus 3.5.9 / langchain4j 1.9.0 / SDN 7.x / Seata 2.x / 无 native 绑定——**全部 JDK 21 安全，无阻塞项**
- **虚拟线程接管盲区（关键陷阱）**：`AiService.java:249` 的 `CompletableFuture.runAsync(...)` 跑在 `ForkJoinPool.commonPool`，`spring.threads.virtual.enabled` **接管不到它**——这是 SSE 流式问答真正阻塞的线程，不改这处则虚拟线程对 AI 场景零收益
- **切换阻力**：项目无自定义 `@Async`/`Executor` Bean（仅 industry-chain 有一个手动 `ThreadPoolTaskExecutor`，AI 有一个单线程索引重建池），不冲突

## 分三层推进：先修快赢项 → 升 JDK 21 → 开虚拟线程

### 第一层：修配置矛盾与浪费（0 风险，立即腾资源）

| # | 文件 | 问题 | 改动 |
|---|------|------|------|
| 1a | `backend/docker/redis/redis.conf:6` | `maxmemory 1gb` 但容器 limit 仅 256M → 必被 OOM kill | `maxmemory 1gb` → `maxmemory 200mb` |
| 1b | `backend/docker/mysql/my.cnf:23-28` | `buffer_pool 4G / log_file 1G` 是死代码（实际生效 512M/512M），误导优化者 | 注释标注"实际值由 docker-compose command 注入，此文件在 Windows 不生效"，或直接改为与实际一致的 512M 避免误导 |
| 1c | `docker-compose.yml` | kafka / nacos / seata 三个服务**无 `deploy.resources.limits`**，吃内存无 cgroup 约束 | 给三者加 limit：kafka `cpus:1.0/mem:768M`、nacos `cpus:0.5/mem:640M`、seata `cpus:0.5/mem:512M`（nacos 已设 JVM_XMX=512m，limit 留余量） |
| 1d | `docker-compose.yml` | 5 个 Java 服务缺 `MALLOC_ARENA_MAX=2`（仅 graph 设了，省显著 off-heap） | 给 user/trade/ai/industry-chain/gateway 补 `MALLOC_ARENA_MAX: "2"` 环境变量 |
| 1e | `backend/Dockerfile` ENTRYPOINT | 无 GC 日志、无 `-XX:+UseContainerSupport` 显式声明、无 `MaxRAMPercentage` | 见第二层一起改（升 JDK 时顺带加 GC 日志参数） |

**预期收益**：消除 Redis OOM 风险、堵住 3 个无 limit 服务的内存无底洞、off-heap 内存下降。

### 第二层：JDK 17 → 21（兼容性已验证安全）

| # | 文件 | 改动 |
|---|------|------|
| 2a | `backend/pom.xml:32` | `<java.version>17</java.version>` → `21` |
| 2b | `backend/scripts/mvn17.ps1:13` | `maven:3.9-eclipse-temurin-17` → `-21`（脚本可留名 mvn17 或重命名） |
| 2c | `backend/Dockerfile:3` | build stage `maven:3.9-eclipse-temurin-17` → `-21` |
| 2d | `backend/Dockerfile:46,53,59,66,72,78` | 6 个 runtime stage `eclipse-temurin:17-jre` → `21-jre` |
| 2e | `backend/Dockerfile` ENTRYPOINT（顺带） | 给每个服务加 `-XX:+UseZGC` 或保持 G1（JDK21 G1 已够好，倾向保持 G1 不动 GC）+ `-Xlog:gc*:file=/app/gc.log:time,uptime,level,tags:filecount=2,filesize=10m`（GC 日志，便于排查） |
| 2f | `Dockerfile`（根，旧单jar） | `temurin-17` → `21`（保持一致，虽然可能已不用） |

**验证**：`mvn -B package` 全量编译通过 + 6 服务本地 `docker compose up` 启动正常 + 冒烟（登录→提问→支付链路）。

### 第三层：开虚拟线程（核心目标）

| # | 文件 | 改动 |
|---|------|------|
| 3a | `deploy/nacos/sparrow-common.yaml` | 加 `spring.threads.virtual.enabled: true`（一处生效，所有 5 个 Servlet 服务自动接管 Tomcat 请求线程 / `applicationTaskExecutor` / StreamingResponseBody / SseEmitter 外层） |
| 3b | `backend/sparrow-ai/.../AiService.java:249` ⭐ | **关键修复**：`CompletableFuture.runAsync(task)` → 显式传虚拟线程 executor。新增一个共享 `Executors.newVirtualThreadPerTaskExecutor()`（每次 ask 创建一个即可，虚拟线程极轻量），让 LLM 推理阻塞在虚拟线程而非 commonPool 平台线程 |
| 3c | `backend/sparrow-ai/.../application.yml` | 开虚拟线程后 `server.tomcat.threads.max` 失效，但需确认 `max-connections`（默认 8192）够支撑并发 SSE 长连接——保持默认即可 |
| 3d | `backend/sparrow-graph/.../application.yml` | 注释更新：`server.tomcat.threads.max:150` 开虚拟线程后失效（保留配置无害，但加注释说明） |
| 3e | industry-chain `IndustryChainAsyncConfig` | **暂不动**（手动 ThreadPoolTaskExecutor core=2/max=4，虚拟线程开关不改它）。若用户后续想统一，再单独评估。在 SKILL/注释里记录此盲区 |

**3b 是整个计划的技术核心**：`spring.threads.virtual.enabled=true` 让 Tomcat 用虚拟线程接收请求，但请求进来后 `askStream` 立即 `CompletableFuture.runAsync` 把阻塞工作甩给 commonPool——那个 commonPool 还是平台线程，LLM 的 60s 阻塞仍然占用平台线程。必须改成虚拟线程 executor，SSE 场景的收益才真正落地。

**同步评估连接池天花板**：虚拟线程放开 SSE 并发后，并发请求数上探，但 HikariCP=50 / Lettuce=100 是硬上限。**本计划暂不上调**（MySQL 侧 1500 连接余量充足，但先验证虚拟线程实际收益，避免一次改太多）。在计划里记录"若压测显示连接池成瓶颈，HikariCP 可上调到 80-100"。

## 验证方式
- 第一层：`docker compose config` 校验 + 启动观察内存
- 第二层：`mvn package` 编译 + 冒烟测试
- 第三层：本地多并发 SSE 请求验证（虚拟线程下并发流式问答不耗尽线程）

## 风险
- 主要风险在第二层 JDK 升级，但摸底已确认所有依赖 JDK 21 安全、无 native 绑定、Seata 使用面极窄（1 处）。
- 第三层虚拟线程的"synchronized 块导致 pinning"风险：JDK 21 已大幅缓解（JEP 491 在 24 才彻底修，21 仍有少量 pinning 点）。若压测发现 pinning，降级方案是回退 `spring.threads.virtual.enabled=false`（一行配置，无代码侵入，除 3b 外）。

按"先 1 → 再 2 → 最后 3"顺序实施，每层独立可验证、可回滚。
