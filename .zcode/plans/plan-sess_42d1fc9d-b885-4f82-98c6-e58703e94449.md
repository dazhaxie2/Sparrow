## 三个 Bug 的根因与修复计划

### Bug 1: 切换标签页后产业链调研状态丢失

**根因**：`App.vue` 的 `<keep-alive>` 包住了 `<router-view>`,但 `IndustryChainWorkbenchView.vue` 只注册了 `onMounted`/`onUnmounted`,**没有 `onActivated`/`onDeactivated`**。在 keep-alive 下,离开页面时 `onUnmounted` 不触发(组件被缓存而非销毁),SSE `fetch` 流变成游离态——浏览器可能断开空闲连接;返回页面时又没有 `onActivated` 去重连或重新 `load()`。于是 `forumEvents` 停止更新、若实例被驱逐则 `detail` 归空,表现为「调研不见了」。

**修复**(单文件、治本):`IndustryChainWorkbenchView.vue`
- 新增 `onActivated` → 若仍在 `researching` 则重新 `connectEvents()`(且先 `stopEvents()` 清理可能失效的旧流);否则若 `detail` 为空则 `load()`。
- 新增 `onDeactivated` → `stopEvents()`(断开 SSE 流、停轮询),避免游离连接。返回时由 `onActivated` 恢复。
- `onUnmounted(stopEvents)` 保留,兜底真实销毁场景。

### Bug 2: Agent 动态思考过程看不到(两者都做)

**现状**：后端用阻塞式 `ChatModel.chat()`,只在 6 个固定节点发 `progress`,每轮 Agent 总结完成后才发一条 `forum`。看不到逐 token 思考。

**修复**:
- **(a) 细粒度阶段事件** —— 在每个 Agent 的检索/首轮总结/反思/深挖总结步骤开始时,通过新的 `IndustryChainEventHub.thinking(cardId, runId, role, message)` 发 `thinking` 事件(如「行业 Agent · 反思第 2 轮 · 检索中」)。`Orchestrator` 的 planning/evidence/mapping/writing 各阶段也补发 `thinking`。前端 `connectEvents` 增加 `thinking` 分支 → 追加到 `forumEvents`(source=`THINKING`),让论坛连续滚动而非跳变。
- **(b) 流式 token** —— `ChatModelProvider` 新增 `streamingModel()` 与流式构建(用 langchain4j 的 `OpenAiStreamingChatModel` + `StreamingChatModel`)。Agent 的 `summarize`/`reflect` 改为流式回调:首个 token 前发一条「…正在思考」thinking 事件,token 增量通过 `thinking` 事件实时推送(节流,如每 200ms 一批)。`IndustryChainAiConfig.buildFromProps`/`buildFrom` 同时构建阻塞模型(规划/证据/图谱/报告等需要完整 JSON 的路径)与流式模型(Agent 文本生成路径)。前端 `AgentForum` 的 `THINKING` 气泡改为「增量更新同一条」(用 role+round 做幂等 key),呈现打字机效果。

### Bug 3: 调研速度极慢(四项全做)

1. **并行化联网检索与富化** —— `WebSearchClient.search`:把 `searchOne(query)` 多查询改为 `CompletableFuture` 并行 + 去重;`enrich(candidate)` 逐条串行改为并行。新增共享检索:3 个 Agent 的首轮检索只跑一次、结果共享(在 `Orchestrator` 层做一次首轮共享检索后传给各 Agent 的后续反思深挖)。纯网络并行,不碰 LLM 逻辑。
2. **降低超时与 maxTokens 分级** —— `IndustryChainAiConfig`:默认 timeout 180s→45s、maxRetries 2→1;`maxTokens` 不再统一 3000,改为按调用分级(反思/检索总结 800、报告主体 3000)。`IndustryChainProperties`/`application.yml` 同步默认值。
3. **扩大异步线程池** —— `IndustryChainAsyncConfig`:corePoolSize 2→4、maxPoolSize 4→8,让 3 个 Agent 完全并行并容纳并发调研。
4. **精简反思轮次** —— `IndustryChainResearchAgent.MAX_REFLECTIONS` 2→1,直接砍掉每个 Agent 2 次 LLM 调用(3 Agent 共省 6 次)。

### 实施顺序与验证
1. **Bug 1**(前端、低风险、独立)→ 先做,`vite build` + 浏览器实测切换图谱⇄产业链。
2. **Bug 3 的 2/3/4 项**(后端配置/小改、低风险)→ 提速立竿见影,`mvn -pl sparrow-industry-chain -am compile` + 启动跑一次调研看时长。
3. **Bug 3 的第 1 项**(WebSearchClient 并行化 + 共享检索)→ 中等改动,编译 + 跑调研确认来源数量不减少。
4. **Bug 2(a) 细粒度 thinking 事件** → 后端 `EventHub.thinking` + Agent/Orchestrator 发事件 + 前端渲染,编译 + 浏览器看论坛连续滚动。
5. **Bug 2(b) 流式 token** → 改动最大最后做;`ChatModelProvider` 增流式模型、Agent 文本生成走流式、前端 thinking 气泡增量更新。
6. 每步后跑 `node tools/harness.mjs guard` + 对应 build。

### 风险
- 流式改造(Bug 2b)涉及 langchain4j 的 `StreamingChatModel` API 与回调线程模型,需确认依赖版本支持(本仓库已用 langchain4j)。若该步骤踩坑,可单独回退而不影响前 5 步。
- WebSearchClient 并行化需注意搜狗反爬:并行查询数过高可能触发限流;计划用 `CompletableFuture.allOf` 限并发(如 3 路)而非全开。
- 降低 timeout 到 45s 后,少数慢调用可能失败回退;`chatOr` 已有 fallback 路径,可接受。
