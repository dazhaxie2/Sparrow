# 本地启动 Sparrow 全栈 + 完备测试方案

## 阶段一：全容器化降配启动（保留现有数据）

**命令**：`docker compose -f docker-compose.yml -f docker-compose.local.yml up -d --build`

执行步骤：
1. 先 `docker compose ps` 检查现有容器/卷状态；若 sparrow_industry_chain 库缺失，按 README 提示补迁移脚本（不执行 `down -v`）
2. 启动 11 中间件 + 6 业务服务，后台运行，轮询健康状态
3. 逐个确认 6 服务通过 Nacos 注册成功、gateway:8080 可达
4. **预期**：首次构建 20-40 分钟（下载依赖+打 6 镜像），后续启动分钟级
5. 验收点：`curl http://localhost:8080/api/graph/tree` 返回 code:0 且 nodes≥77

## 阶段二：运行现有测试基线

按依赖顺序跑，建立"修改前全绿"基线：
1. `mvn test`（本地 JDK21）— 现有 7 个 JUnit 单测
2. `frontend/ npm run guard:architecture` — 架构边界守卫
3. `backend/scripts/smoke.ps1` — 11 步端到端冒烟（注册→图谱→AI规则→下单→支付幂等→会员解锁）
- 任一失败先修复，确保基线干净再继续

## 阶段三：新增 Java 单测补缺口（纯 Mockito，与现有风格一致）

**sparrow-common**（需先在 pom.xml 加 `spring-boot-starter-test` test scope）：
- `ApiResponseTest` — ok()/error() 工厂
- `BizExceptionTest` — code 字段、默认 400、是 RuntimeException
- `TokenKeysTest` / `TopicsTest` — 锁定常量契约（防漂移）
- `UserContextTest` — ThreadLocal 语义、require() 抛 401、@AfterEach clear
- `UserHeaderInterceptorTest` — 头注入、非法头降级匿名
- `GlobalExceptionHandlerTest` — 3 个 handler 分支（含 BindingResult mock）

**sparrow-ai**（pom 已有 test 依赖）：
- `AiServiceTest` — 6 个核心用例：
  - T1 配额耗尽抛 429（在 model 调用前）
  - T2 会员绕过配额（remainingQuota=-1，不碰 Redis）
  - T3 Agent 成功短路（mode=agent，chatModel.chat 不被调用）
  - T4 Agent+RAG 双失败降级 rules（三级降级链，头条用例）
  - T5 classifyApplications：未配置返空 / 配置后解析 JSON 数组过滤无效 id
  - T6 askStream rules 路径事件顺序 meta→delta→done→complete（recording StreamSink + latch）
- `ChatHistoryTest`（如时间允许）— 会话归属校验、title 截断
- `RagIndexerTest`（纯静态）— fingerprint 稳定性/敏感性、chunk 分句

## 阶段四：全接口黑盒测试（新写 PowerShell 脚本，覆盖全部 37 个 /api/** 接口）

**目标**：经 gateway:8080，覆盖每个 /api/** 公共接口。脚本 `backend/scripts/full-api-test.ps1`：

分块测试（每块独立、可单独跑）：
- **user**：register→login→me(401无token)→me(有token)；重复注册失败
- **graph**：tree/overview/nodes(分页)/search/subgraph/clusters/tiles/node/{id}/neighborhood/prerequisites/applications/knowledge-status — 含不存在的 id 返回 404
- **ai**：ask（真实 LLM 走 agent/rag）+ ask/stream（SSE 抓前几帧验证 meta/delta/done）+ sessions CRUD + messages 历史 + DELETE
- **trade**：products 目录 → order 下单 → order/{no} 查询 → orders 列表 → pay/mock/notify（伪造 token 拒绝 + 正常回调 + 重复幂等）
- **chains**：cards CRUD 全套 + messages + runs(启动/查询/取消) + events(SSE) + forum + attachments/upload(PDF)
- 每个接口断言：HTTP 状态 + 业务 code + 关键字段存在；鉴权接口分别测无 token(401) 和有 token

产出 pass/fail 计数报告，失败不中断、最后汇总。

## 阶段五：回归与收尾

1. 全量 `mvn test` 复跑（含新增单测，确认全绿）
2. 重新跑 `full-api-test.ps1` 确认接口稳定
3. 输出最终报告：启动状态 / 接口通过率 / 单测通过数 / 发现的问题清单

## 关键技术决策

| 决策 | 选择 | 理由 |
|---|---|---|
| 启动 | 全容器化+降配 | 你选的；环境隔离，最贴近生产 |
| 数据 | 保留现有卷 | 你选的；避免丢图谱数据，缺库才补迁移 |
| 单测风格 | 纯 Mockito（无 @SpringBootTest） | 与现有 7 个测试完全一致；不引入 Testcontainers，CI/本地都秒级跑 |
| 黑盒脚本 | PowerShell（与 smoke.ps1 同款） | 复用现有 Api() 封装，团队既有约定 |
| controller 层测试 | **不做**（用黑盒脚本覆盖） | controller 是薄转发层，黑盒经 gateway 测更真实，避免为薄层写大量 mock |

## 产出物清单
- `backend/sparrow-common/pom.xml`（加 test 依赖）
- `backend/sparrow-common/src/test/...`（6 个测试类）
- `backend/sparrow-ai/src/test/...`（6-8 个测试类）
- `backend/scripts/full-api-test.ps1`（全接口黑盒）
- 测试报告（终端输出 + 汇总）

## 风险与回滚
- 首次构建慢/OOM：用 local.yml 降配，监控内存；若 OOM 降级到只起核心栈（不含 AI 层）
- 接口测试依赖 AI 真实调用（.env 已配 glm-4.7 key）：AI 接口可能因限流/网络偶发失败，脚本对 AI 部分做软断言（非 0 code 但有降级 answer 即算通过）
- 新增单测若暴露真实 bug：先记录，不强行改产品代码（除非是显然的错误）
