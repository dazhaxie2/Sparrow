# Sparrow 演进路线图

> 对标大厂主干道:**单体快速起步 → 服务化拆分 → 数据架构深化 → 云原生/多活**(见 architecture-evolution-cn.html 的规律 01–06)。
> 目标:1 亿注册用户 · 1000 QPS(均值口径)· 亿级数据量。

---

## 0. 产品定义(功能极其少)

**Sparrow = 科技图**。只做三个功能,每个功能恰好"逼出"一类技术栈:

| 功能 | 描述 | 被逼出的技术 |
|---|---|---|
| 🌳 科技图图谱 | 从石器时代到 AGI 的技术节点 + 依赖边,可视化浏览(类文明系列科技图) | 知识图谱、Redis、MySQL |
| 🤖 AI 向导 | 基于图谱问答:"发明蒸汽机需要哪些前置技术?"、个性化学习路径 | RAG、Agent、Milvus、Kafka |
| 💳 会员支付 | 付费解锁深度内容 / AI 额度,账户体系 + 订单 | Seata、Kafka、分库分表 |

对应大厂规律 05(差异在"被什么逼"):**Sparrow 被支付一致性逼(像微信)+ 被 AI 成本逼(像拼多多)**,这两点决定底座重心。

---

## 1. 容量目标的诚实拆解

先把指标算清楚,避免过度设计:

- **1 亿用户是注册量,不是并发量。** 按 1%–5% DAU 即 100 万–500 万日活;1000 QPS 是均值口径,**架构按峰值 10 倍 = 1 万 QPS 设计**(大促/活动洪峰)。真实 1 亿用户产品峰值远不止于此,但作为演进项目,"均值 1000 / 峰值 1 万"是合理的设计基线。
- **真正上亿行的表只有三类:** 用户表(1 亿)、订单/流水表(亿级)、行为日志(十亿级)→ 这才是分库分表的对象。
- **科技图本身很小**(节点万级、边十万级),是典型**读多写少**数据 → 全量可进 Redis,缓存命中率做到 95%+ 后,图谱浏览几乎不打数据库。
- **AI 是成本瓶颈而非 QPS 瓶颈** → 解法是限流 + 排队(Kafka)+ 语义缓存,而不是堆机器。

## Phase 1 · 单体起步(规律 01:起步都"不讲架构")

**目标:2–4 周跑通产品闭环。增长才是唯一的架构师,这一阶段禁止过度设计。**

### 架构
- **Spring Boot 3 模块化单体**(modular monolith),按 `user / graph / ai / trade` 分包,模块间只通过接口调用——这是为 Phase 2 拆分预埋的缝。
- **MySQL 8 单实例**:用户、订单、科技图节点表 + 边表(邻接表建模,`tech_node` / `tech_edge`)。
- **Redis 单机**:科技图子图缓存(整棵子树序列化为 JSON)、会话。
- **Milvus Standalone**:科技节点描述向量化,最朴素的 RAG——检索 top-k 节点喂给 LLM 回答。
- **支付**:支付宝/微信沙箱,单库本地事务即可,**此时不需要 Seata**。
- **前端**:AntV G6 / ECharts graph 渲染科技图(力导向 + 层级布局)。
- **Docker Compose 一键起全套**(app + mysql + redis + milvus)。

### 明确不做
微服务、Kafka、Seata、Neo4j、K8s——单体阶段引入它们只有成本没有收益。

### 验收
- 科技图可浏览、AI 可问答、沙箱支付可下单
- 单机压测 50 QPS 稳定,P99 < 200ms

## Phase 2 · 服务化拆分(规律 02:增长拐点逼出服务化)

**触发点:模块耦合开始互相拖累——AI 流量(长耗时、高成本)和支付流量(强一致、低延迟)特征迥异,必须隔离。**

### 服务拆分(5 个服务,不要更多)
```
                        ┌─────────────┐
                        │   Gateway    │  Spring Cloud Gateway:路由/鉴权/限流
                        └──────┬──────┘
        ┌──────────┬───────────┼───────────┬──────────┐
   ┌────┴───┐ ┌────┴───┐ ┌────┴────┐ ┌────┴───┐
   │  user  │ │ graph  │ │   ai    │ │ trade  │
   │ 用户·会员│ │ 科技图  │ │RAG·Agent│ │ 订单·支付│
   └────────┘ └────────┘ └─────────┘ └────────┘
```

`notify` 不独立成服务。通知、embedding 重建、对账等异步能力先通过 Kafka topic 和各服务内消费者承接,避免 Phase 2 早期拆出第 6 个服务。

### Phase 2 里程碑切分

- **M1 服务化骨架**:多 Maven module、Nacos 注册/配置、Gateway 统一鉴权、OpenFeign 裸调用;仍连接单库 `sparrow`。
- **M2 事务收口**:拆分 user/graph/ai/trade 业务库,引入 Seata AT,验证支付链路回滚。
- **M3 事件驱动**:引入 Kafka,支付成功、图谱变更、embedding 同步走事件,消费者必须幂等。
- **M4 图数据库**:图谱从 MySQL 邻接表迁移 Neo4j,多跳路径查询改为 Cypher。
- **M5 Agent 与治理**:LangChain4j Agent 工具调用,接 Sentinel 做 AI 限流熔断。

### 中间件引入(每个都有明确理由)
- **Nacos**:注册中心 + 配置中心;**OpenFeign / Dubbo**:服务间 RPC;**Sentinel**:接口级限流熔断(AI 接口必须限流)。
- **Kafka 进场**:① 订单事件(支付成功 → 开通会员 → 发通知,事件驱动解耦);② 行为日志异步落盘;③ **内容变更 → embedding 流水线**(图谱编辑后自动重算向量进 Milvus)。
- **Seata 进场**:下单 = trade 扣款 + user 开通会员/扣 AI 额度,跨服务事务。**AT 模式起步**(侵入小),支付回调对账走消息最终一致。
- **知识图谱升级**:MySQL 邻接表多跳查询(递归 CTE)在 3 跳以上撑不住 → 迁移 **Neo4j / NebulaGraph**,支持"这项技术的全部前置依赖链"类查询。
- **Agent 成型**:从"单次 RAG"升级为工具调用 Agent——工具集 = 查图谱(Cypher)/ 查向量(Milvus)/ 查用户进度,LLM 自主编排多步推理。
- **Redis 加码**:分布式锁(防重复下单)、接口限流计数、热点节点缓存。
- **部署**:Docker 镜像 + K8s 单集群(minikube/k3s 起步即可)。

### 验收
- 1000 QPS 压测达标(图谱读为主),P99 < 300ms
- 杀掉任意一个服务实例,系统可用(验证注册中心摘除 + 重试)
- 支付链路:Seata 回滚演练通过,Kafka 事件不丢不重(幂等消费)

## Phase 3 · 数据架构深化(规律 03:通用方案"不够用")

**触发点:用户表破千万、订单表破亿、单表查询劣化;活动洪峰打挂同步链路。这是模拟"1 亿用户"的核心阶段。**

### 分库分表(对标 PDD 电商主干)
- **ShardingSphere**:用户表按 `uid hash` 拆 **16 库 × 64 表**(单表 ~10 万行 × 1024 = 亿级容量);订单表按 `uid` 分片 + **基因法**(订单号嵌入 uid 后 N 位,买家查单不跨片)。
- **读写分离**:MySQL 主从,读流量走从库;**冷热分离**:一年前订单归档到历史库。
- 数据造数:写脚本灌 1 亿行模拟数据,**所有 SQL 在亿级数据上验证执行计划**——这是"亿级数据量"目标的真正落点。

### 削峰与数据管道(Kafka 集群化)
- **削峰**:会员秒杀/大促 → 请求进 Kafka 排队,trade 服务按消费能力处理(对标 PDD"消息队列削峰")。
- **CDC 管道**:Canal 监听 MySQL binlog → Kafka → 同步 ES(搜索)/ Milvus(向量)/ 数仓,业务代码零侵入。

### 支付走向成熟(对标微信"强一致被支付逼出来")
- 账户体系:余额/额度账户 + 流水表,**对账系统**(日终核对订单流 vs 账务流)。
- 事务策略收敛:**Seata 只留核心强一致场景(扣款开通会员),其余链路降级为事务消息/本地消息表最终一致**——大厂的真实取舍就是"能最终一致就不强一致"。

### RAG/Agent 深化
- **GraphRAG**:向量检索(Milvus)+ 图谱多跳(Neo4j)混合召回 → rerank → 生成,回答"路径类"问题质量碾压纯向量 RAG。
- **语义缓存**:问题向量相似度 > 阈值直接返回缓存答案,LLM 成本砍 50%+。
- AI 请求全部过 Kafka 排队 + 令牌桶限流,成本可控。

### 可观测性(没有它压测就是盲人摸象)
- SkyWalking 全链路追踪 + Prometheus/Grafana 指标 + ELK 日志。
- **全链路压测**:影子表方案,验证峰值 1 万 QPS。

### 验收
- 亿级数据下核心查询 P99 < 100ms(缓存命中)/ < 500ms(穿透到库)
- 峰值 1 万 QPS 压测通过;Kafka 积压可追平
- 对账零差错;任意服务实例宕机不丢单

## Phase 4 · 云原生·多活(规律 04:终点收敛到云原生)

**对个人项目而言这阶段是"演练价值">"实用价值",目标是把多活的形搭出来、把故障演练做扎实。**

- **K8s 生产化**:HPA 弹性伸缩(AI 服务按 GPU/队列深度扩缩)、滚动发布、资源配额。
- **Service Mesh(Istio)**:流量治理从 SDK 下沉到 Sidecar,灰度发布、故障注入。
- **单元化演练**:按 uid 划分单元,用 K8s 多 namespace 模拟"两个机房",验证单元封闭(用户请求不跨单元)+ 单元故障切流(对标小红书单活→双活→多活三级跳)。
- **混沌工程**:ChaosBlade 随机杀 Pod / 注入网络延迟,验证柔性可用(对标微信"海量服务之道":过载保护、基于不可靠硬件构造可靠系统)。
- **Serverless 化尝试**:AI 推理用 Knative 按需弹零。

## 3. 技术栈进场时机一览

| 技术 | Phase 1 | Phase 2 | Phase 3 | Phase 4 |
|---|---|---|---|---|
| **MySQL** | 单库本地事务 | 读写分离 | 分库分表 16×64 + 冷热分离 | 容器化/Operator |
| **Redis** | 单机缓存 | 分布式锁、限流 | Cluster + 多级缓存(Caffeine+Redis) | — |
| **Kafka** | ✗ 不引入 | 事件驱动 + embedding 流水线 | 削峰 + CDC 管道、集群化 | — |
| **Seata** | ✗ 本地事务够用 | AT 模式跨服务事务 | 收敛到核心场景,余者最终一致 | — |
| **Milvus** | Standalone 朴素 RAG | 独立服务 + Agent 工具 | GraphRAG 混合召回 + 语义缓存 | 分布式版 |
| **知识图谱** | MySQL 邻接表 | 迁移 Neo4j 多跳查询 | 与向量混合召回(GraphRAG) | — |
| **RAG/Agent** | 单次检索问答 | 工具调用 Agent | 多路召回 + rerank + 成本治理 | Serverless 推理 |
| **Docker** | Compose 一键起 | 镜像 + K8s 单集群 | 全链路压测环境 | Mesh + 单元化 + 混沌 |

每一行都遵循同一个纪律:**晚于需要时引入 = 技术债;早于需要时引入 = 过度设计。** 进场时机本身就是演进项目最有讲头的部分。
